2. Основные сущности системы
2.1. Ценовой формат (ЦФ)
Сущность, определяющая правила формирования цен для группы контрагентов. Содержит:
Поле	Описание
Наименование	Название ценового формата
Код	Уникальный идентификатор
Филиал	Привязка к конкретному филиалу/складу
Правило ЦО	Применяемое правило ценообразования
Прогиб	Величина отступа от цены конкурента в процентах (напр., 0,3%)
Таблица наценок	Диапазоны себестоимости и соответствующие наценки

2.2. Прайс-лист (ПЛ)
Результирующий документ с рассчитанными ценами. Содержит: список товаров, рассчитанные цены, дату формирования, дату активации, пользователя-автора. Для каждой аптеки/контрагента может формироваться свой прайс-лист.
2.3. Прайс-листы конкурентов (ПЛК)
Данные о ценах конкурентов, получаемые из внешних источников. Содержат:
•	Поставщик/конкурент (Фармцентр, Видман, Провизор, Фармнавигатор).
•	Дата актуальности цен.
•	Регион (при отсутствии региональных данных — эталонный алматинский прайс).
•	Источник данных: API ph.center, Excel-файлы, парсинг сайтов.

Важно: цены конкурентов на ph.center указываются с учетом НДС в зависимости от группы лекарственных средств (5% или 16%)
2.4. Универсальные списки
Списки с параметрами для отдельных товаров, позволяющие переопределить стандартную логику ценообразования:
Тип списка	Описание
Фиксированная цена	Цена задаётся вручную, формула не применяется
Минимальная цена	Цена не может быть ниже указанного значения
Максимальная наценка	Наценка не может превышать указанный процент
Государственная цена	Предельная цена(модельная цена не может быть выше чем гос)

Каждый список имеет статус (активен/не активен), дату начала и окончания действия.
2.5. Контрагенты и категории клиентов
Справочник клиентов из SAP содержит: код, наименование, холдинг. Клиенты категоризируются по объёму товарооборота (формула задаётся в SAP). Категория клиента напрямую влияет на цену: чем выше категория — тем ниже цена. Также учитывается отсрочка платежа: чем больше отсрочка — тем выше цена.
2.6. Товары (SKU)
Товарный справочник содержит: наименование, артикул/код, себестоимость, рейтинг, остатки по складам, сроки годности. Необходимо учитывать ситуацию, когда на 1 физический товар может приходиться до 3 карточек в SAP (дубли). 

3. Логика ценообразования
3.1. Таблица наценок
Наценка определяется на основании диапазона учётной себестоимости товара:
Диапазон себестоимости (тенге)	Наценка
0 — 49,99	10%
50 — 99,99	8%
100 — 499,99	5%
500 — 999,99	4%
1 000 — 2 999,99	3%
3 000+	1%

Диапазоны и проценты наценок должны быть настраиваемыми через интерфейс администратора.
3.2. Формулы расчёта
3.2.1. Минимальная допустимая цена (МДЦ)
МДЦ = Себестоимость × (1 + Рекомендованная наценка%)
Рекомендованная наценка определяется по таблице наценок из п. 3.1 на основании себестоимости товара. МДЦ является нижней границей цены — итоговая цена не может быть ниже этого значения.
3.2.2. Приведённые цены конкурентов и ранжирование
Для каждого товара собираются цены из всех привязанных ПЛК (от 1 до 5 конкурентов). Каждая цена приводится к единой базе:
Приведённая цена конкурента = Цена из ПЛК × Коэффициент
Коэффициент задаётся для каждого источника ПЛК и позволяет учитывать разницу в условиях (НДС, тип цены и т.д.).
После приведения все цены конкурентов сортируются по возрастанию. Из отсортированного списка выделяются:
• Ц1 — 1-я минимальная цена конкурента (самая низкая);
• Ц2 — 2-я минимальная цена конкурента;
• Ц3 — 3-я минимальная цена конкурента.
Если конкурентов менее трёх, используются только доступные значения (например, при двух конкурентах — Ц1 и Ц2).
3.2.3. Цена с прогибом
Для каждой из ранжированных цен конкурентов (Ц1, Ц2, Ц3) рассчитывается цена с прогибом:
Цена с прогибом = Цена конкурента × (1 − Прогиб%)
Прогиб — величина в процентах, задаётся в настройках ценового формата. Цена с прогибом позволяет выставить свою цену чуть ниже цены конкурента на заданный процент.
3.2.4. Каскадный подбор итоговой цены
Система последовательно пытается встать с прогибом под цену конкурента, начиная с самой низкой. Условие: цена с прогибом не должна быть ниже МДЦ (минимальной допустимой цены). Алгоритм:
1. Рассчитать цену с прогибом от Ц1: П1 = Ц1 × (1 − Прогиб%). Если П1 ≥ МДЦ → Итоговая цена = П1.
2. Если П1 < МДЦ, рассчитать цену с прогибом от Ц2: П2 = Ц2 × (1 − Прогиб%). Если П2 ≥ МДЦ → Итоговая цена = П2.
3. Если П2 < МДЦ, рассчитать цену с прогибом от Ц3: П3 = Ц3 × (1 − Прогиб%). Если П3 ≥ МДЦ → Итоговая цена = П3.
4. Если ни под одну из трёх минимальных цен встать не удалось (все ПN < МДЦ), либо цены конкурентов отсутствуют то идем по более высокой цене(товар имеется только у нас).

Таким образом, система всегда стремится предложить максимально конкурентную цену, но не ниже минимальной допустимой наценки.
3.2.5. Пример расчёта
Исходные данные: себестоимость = 100 тенге, рекомендованная наценка = 10%, прогиб = 0,3%. Цены конкурентов: 105, 115, 112, 120, 130 тенге.
1. МДЦ = 100 × (1 + 10%) = 110 тенге.
2. Ранжирование цен: Ц1 = 105, Ц2 = 112, Ц3 = 115.
3. Попытка 1: П1 = 105 × (1 − 0,3%) = 104,69 тенге. П1 < МДЦ (104,69 < 110) → не подходит.
4. Попытка 2: П2 = 112 × (1 − 0,3%) = 111,66 тенге. П2 ≥ МДЦ (111,66 ≥ 110) → подходит.
Итоговая цена = 111,66 тенге.
3.3. Применение ограничений (универсальные списки)
Ограничения применяются только если соответствующий список активен и текущая дата входит в период его действия.
3.3.1. Приоритет применения
Ограничения применяются в следующем порядке приоритета (от высшего к низшему):
1.	Фиксированная цена — полностью заменяет расчётную цену.
2.	Фиксированная наценка — Цена = Себестоимость × (1 + фикс. наценка).
3.	Ограничения (мин./макс.) — цена корректируется в рамках допустимого коридора.
4.	Базовая логика — стандартный расчёт по формулам п. 3.2.
3.3.2. Государственная предельная цена
Для товаров с установленной государственной ценой: итоговая цена не может превышать предельную цену согласно действующему приказу. Файл с предельными ценами загружается из Excel. При изменении приказа (если новая предельная цена ниже текущей отпускной) запрашивается компенсация у поставщика и устанавливается новая цена.
3.4. Управление сроками годности
Система должна поддерживать управление товарами с приближающимся сроком годности и неликвидными товарами.
3.4.1 Определение условий
Товар считается приближающимся к окончанию срока годности, если до даты истечения остается менее N дней (значение настраивается, например 60 дней).
4. Полный алгоритм расчёта цены
Для каждого товара в прайс-листе выполняются следующие шаги:
5.	Загрузить остатки и себестоимость из SAP.
6.	Определить наценку по диапазону себестоимости (таблица наценок).
7.	Рассчитать базовую цену: Себестоимость × (1 + Наценка).
8.	Получить цены конкурентов из привязанных ПЛК, привести к единой базе (× коэффициент согл,формуле). 
9.	Отсортировать цены конкурентов по возрастанию, выделить Ц1, Ц2, Ц3 (1-ю, 2-ю, 3-ю минимальные). При отсутствии цен конкурентов, выбираем более высокую.
10.	Каскадный подбор итоговой цены: последовательно попытаться встать с прогибом под Ц1, Ц2, Ц3 при условии ≥ МДЦ. Если ни одна не подходит — итоговая цена = МДЦ.
11.	Проверить и применить универсальные списки (фикс. цена, фикс. наценка, мин./макс.).
12.	Проверить соответствие государственной предельной цене.
13.	Выполнить округление (два знака после запятой пр.115,898 – 115,90) по заданным правилам.
14.	Сохранить результат и зафиксировать в логе.
5. Интеграции с внешними системами
5.1. SAP ERP
Интеграция с SAP является ключевой для работы системы:
•	Импорт: справочник товаров, себестоимость, остатки по складам, сроки годности.
•	Импорт: справочник контрагентов, категории клиентов (по товарообороту).
•	Экспорт: загрузка рассчитанных Excel-файлов с ценами обратно в SAP.
SAP отдаёт цены в ph.center. Категории задаются в SAP 
5.2. Фармцентр (ph.center)
Основная торговая площадка, на которой выкладываются прайс-листы всех дистрибьюторов:
•	Получение цен конкурентов через API ph.center.
•	Публикация собственных прайс-листов.
•	Учёт: цены на ph.center указываются с учетом НДС, в учётной системе — без НДС.
•	Учёт автоматических скидок, генерируемых сайтом.
Необходимо изучить и документировать доступные методы API ph.center.
5.3. Сайты конкурентов
Для конкурентов, не представленных на ph.center, данные получаются через авторизованный доступ к сайтам. Имеются аптечные и дистрибьюторские логины. Каждый сайт требует авторизации. Нужно реализовать парсинг с учётом авторизации для получения прайс-листов, остатков и сроков годности.
5.4. Обмен файлами
Ряд агрегаторов (напр., Видман) высылают прайс-листы в формате Excel. Система должна поддерживать загрузку и автоматический разбор таких файлов.


6. Функциональные требования
6.1. Модуль «Ценовые форматы»
•	Список всех ценовых форматов с фильтрацией по филиалу.
•	Создание и редактирование ЦФ: наименование, код, филиал, правило ЦО, прогиб.
•	Настройка таблицы наценок (диапазоны и проценты).
•	Привязка источников ПЛК к ценовому формату.

6.2. Модуль «Прайс-листы»
•	Список всех прайс-листов с фильтрацией по дате, статусу, филиалу.
•	Создание нового ПЛ: выбор ЦФ, запуск расчёта, предварительный просмотр.
•	Просмотр детализации ПЛ: для каждого SKU — себестоимость, базовая цена, цена конкурента, итоговая цена, применённые ограничения.
•	Массовый пересчёт при изменении входных данных.
•	Экспорт в Excel для загрузки в SAP.
•	Генерация отдельных ПЛ для товаров с истекающими сроками годности.
6.3. Модуль «Прайс-листы конкурентов»
•	Автоматическое получение ПЛК из ph.center через API.
•	Загрузка ПЛК из Excel (для Видман и др.).
•	Парсинг сайтов конкурентов через авторизацию.
•	Отображение ПЛК с фильтрацией по конкуренту, дате, региону.
•	Автоматическое сопоставление SKU (мэтчинг товаров) с возможностью ручной корректировки.
6.4. Модуль «Универсальные списки»
•	Создание и редактирование списков по типам (фикс. цена, фикс. наценка, мин. цена, макс. наценка, гос. цена).
•	Загрузка списков из Excel.
•	Управление статусом и периодом действия.
•	Привязка к товарам.
6.5. Модуль «Сопоставление SKU»
Ключевой модуль для решения проблемы 50%-ного несовпадения товарных позиций:
•	Автоматический мэтчинг SKU по наименованию, производителю, дозировке, форме выпуска.
•	Интерфейс ручного сопоставления для позиций, не найденных автоматически.
•	Учёт дублей в SAP (до 3 карточек на 1 товар).
•	История сопоставлений для переиспользования.
6.6. Модуль «Аналитика»
6.6.1. Метрики ценовой позиции
Метрика	Сокр.	Описание
Левое плечо	ЛП	Цена ниже конкурента
Зона логичности	ЗЛ	Цена выше конкурента на 0–3%
Правое плечо	ПП	Цена выше конкурента более чем на 3%
Непересекаемый	—	Нет данных по конкуренту для сравнения

6.6.2. Отчёты
•	По каждому ПЛ: количество SKU в каждой категории (ЛП/ЗЛ/ПП/непересекаемый), доля в %.
•	Прогноз vs факт: сравнение ожидаемого распределения с фактическим.
•	Динамика изменений цен по времени.
•	Дашборд с ключевыми метриками на главной странице.
•	Графики и таблицы с возможностью фильтрации и экспорта.
6.7. Модуль «Настройки»
•	Настройка таблиц наценок (диапазоны и проценты).
•	Настройка прогиба (по умолчанию и по ЦФ).
•	Выбор правила ценообразования.
•	Управление пользователями и правами доступа.
•	Возможность включения/отключения категоризации клиентов при расчёте цен.

9. Структура данных (минимальная)
Основные сущности базы данных:
•	products — товары (SKU, наименование, производитель, форма, дозировка).
•	product_costs — себестоимость (товар, филиал, дата, сумма).
•	product_stocks — остатки (товар, склад, количество, срок годности).
•	price_formats — ценовые форматы (код, филиал, правило, прогиб).
•	markup_ranges — диапазоны наценок (ЦФ, от, до, процент).
•	price_lists — прайс-листы (ЦФ, дата, статус, автор).
•	price_list_items — позиции ПЛ (ПЛ, товар, себестоимость, базовая цена, цена конкурента, итоговая цена).
•	competitor_prices — цены конкурентов (конкурент, товар, цена, дата, регион, источник).
•	universal_lists — универсальные списки (тип, статус, период).
•	universal_list_items — элементы списков (список, товар, значение).
•	contractors — контрагенты (код, наименование, холдинг, категория, рейтинг).
•	sku_mappings — сопоставления SKU (свой SKU, SKU конкурента, метод мэтчинга).
•	price_change_log — лог изменений цен (товар, старая цена, новая цена, дата, причина).
•	ratings - числовой показатель, отражающий надежность или приоритет контрагента.
10. Нефункциональные требования
10.1. Производительность
•	Расчёт прайс-листа на 1 250+ SKU — не более 30 секунд.
•	Массовый пересчёт всех ПЛ — не более 5 минут.
•	Загрузка и парсинг Excel-файла до 10 000 строк — не более 10 секунд.
10.2. Надёжность
•	Логирование всех изменений цен с указанием причины и источника. Лог(комментарий, примечание) формирования цены (что это за цена, с учетом Гос цены, под конкурента 1 или 2, ПДЦ и тд)
•	Возможность отката к предыдущей версии прайс-листа.
•	Автосохранение промежуточных результатов расчёта.
10.3. Масштабируемость
•	Система должна обеспечивать горизонтальное масштабирование и стабильную работу при росте количества SKU.
•	Поддержка 10+ конкурентов.
•	Поддержка нескольких филиалов и складов.
10.4. Безопасность
•	Ролевая модель доступа (администратор, менеджер, аналитик).
•	Хранение логинов/паролей к сайтам конкурентов в зашифрованном виде.
•	Аудит действий пользователей.
10.5. Совместимость
